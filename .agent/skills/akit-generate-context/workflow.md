# Generate Project Context — AI-Optimized Rules

**Goal:** Create a concise `.agent/project-context.md` capturing critical rules, patterns, and conventions that AI agents must follow when working on this project.

**Your Role:** You are a technical facilitator discovering the "unobvious" rules that LLMs need reminding of. Focus on things that are NOT obvious from reading the code alone.

---

## LANGUAGE CONFIGURATION

Read `.agent/config.yaml` and find:
- `communicationLanguage` — the language to use
- `responseStyle` — the interaction style

### Response Style Guide
- **technical** → Concise, code-focused senior peer. Minimal explanation, maximum code.
- **casual** → Friendly companion. Uses emoji, simple explanations, encouraging tone.
- **formal** → Structured mentor. Detailed explanations, step-by-step guidance, thorough.

✅ YOU MUST communicate in `{communicationLanguage}` at all times.
✅ All output, guidance, menus, and explanations MUST use `{communicationLanguage}`.

---

## EXECUTION

### Step 1: Discover Project Stack

Scan the project root for these files and extract key information:

| File | Extract |
|------|---------|
| `package.json` | name, scripts, dependencies, engines |
| `tsconfig.json` | target, module, strict settings |
| `.eslintrc*` / `eslint.config.*` | lint rules, plugins |
| `vitest.config.*` / `jest.config.*` | test framework, setup |
| `.prettierrc*` | formatting rules |
| `.gitignore` | ignored patterns |
| `Dockerfile` / `docker-compose.yml` | containerization |
| `.env.example` | environment variables |

**Present findings:** Show the user what was detected, ask for corrections.

### Step 2: Interview for Conventions

Ask the user about each category. Skip if already documented:

1. **Language Rules**
   - Import style (ESM vs CJS, barrel exports?)
   - Naming conventions (camelCase, kebab-case for files?)
   - Error handling pattern (Result type, try/catch, Error classes?)

2. **Framework Rules**
   - Component patterns (functional only? hooks rules?)
   - State management approach
   - API patterns (REST, GraphQL, tRPC?)

3. **Testing Rules**
   - Test file naming (`*.test.ts` vs `*.spec.ts`)
   - Mock patterns (vi.mock, jest.mock, manual?)
   - Coverage requirements

4. **Code Quality**
   - PR/commit conventions
   - Code review requirements
   - Documentation standards

5. **Critical "Don't Miss" Rules**
   - Things that break builds if forgotten
   - Non-obvious environment setup
   - Platform-specific gotchas

### Step 3: Generate project-context.md

Write to `.agent/project-context.md` using this template:

```markdown
# Project Context: {project_name}

> Auto-generated by agent-kit. Edit as needed.

## Tech Stack
- **Language:** {language} {version}
- **Framework:** {framework}
- **Build:** {build_tool}
- **Test:** {test_framework}

## Critical Rules

### Imports
{import_rules}

### Naming
{naming_rules}

### Error Handling
{error_handling_pattern}

### Testing
{testing_rules}

## Don't Forget
{critical_gotchas}

## Project Structure
```
{directory_tree_summary}
```
```

### Step 4: Review with User

Present the generated file for review. Ask:
- "Is anything missing?"
- "Are there rules I got wrong?"
- "Any critical things AI agents keep messing up?"

Update based on feedback.

### Step 5: Store as Memory

Offer to save key rules as agent-kit memories:
```bash
agent memory add --title "Project conventions" --content "{summary}" --type convention
```

This ensures the rules are also available through `agent context`.

### Completion

```
✅ Project context generated: .agent/project-context.md

This file will be used by AI agents to:
• Follow your coding conventions
• Use correct import patterns
• Apply proper error handling
• Run tests the right way

💡 Tip: Update this file whenever conventions change.
   Run /akit-generate-context again to refresh.
```
